
# قوانین پروژه حدیث اسکیت

این قوانین در تمام مراحل توسعه‌ی پروژه اعمال می‌شوند.

## Git و GitHub

1. پس از پایان هر تغییر منسجم و قابل‌بررسی، تغییرات مرتبط را بررسی، تست و commit کن.
2. پس از موفقیت commit، آن را به شاخه‌ی مرتبط در GitHub push کن.
3. برای هر تغییر از پیام commit کوتاه و معنادار با قالب Conventional Commits استفاده کن؛ مانند:
   - `feat: add product listing page`
   - `fix: correct mobile cart layout`
   - `docs: sync design system with Stitch`
4. تغییرات نامرتبط را در یک commit قرار نده.
5. فایل‌های دارای رمز، API key، token یا اطلاعات خصوصی را هرگز commit نکن.
6. قبل از commit، وضعیت Git و diff کامل را بررسی کن و تست‌ها و lint مرتبط را اجرا کن.

> منظور از «هر تغییر»، یک واحد کاری کامل است؛ نه هر ذخیره‌ی فایل یا ویرایش کوچک. این کار تاریخچه‌ی Git را خوانا و قابل بازگشت نگه می‌دارد.

## طراحی و Google Stitch

1. Google Stitch منبع اصلی طراحی رابط کاربری است.
2. پیاده‌سازی باید با `design.md` و فایل‌های پوشه‌ی `stitch/` هماهنگ باشد.
3. پس از تغییر Design System یا اسکرین‌ها، `design.md`، `stitch/DESIGN.md`، `stitch/project.json` و `stitch/screens.json` را همگام کن.
4. رابط فارسی باید RTL، واکنش‌گرا و mobile-first باشد.

## زبان و Graphify

1. ادمین و محتوای پیش‌فرض فارسی است (`i18n` + `localization` در Payload).
2. برای درک ساختار کد، از Graphify استفاده کن (`npm run graphify:update` پس از تغییرات بزرگ).

1. از TypeScript با typeهای صریح استفاده کن و از `any` غیرضروری پرهیز کن.
2. قبل از تحویل، lint، type-check و تست‌های مرتبط باید موفق باشند.
3. کامپوننت‌ها کوچک، قابل استفاده‌ی مجدد و دارای نام‌های روشن باشند.
4. تغییرات را فقط در محدوده‌ی درخواست انجام بده و از refactor نامرتبط پرهیز کن.
5. دسترس‌پذیری، SEO و عملکرد صفحات فروشگاه را در هر تغییر UI در نظر بگیر.
